# Plan Tunneling
## 前置条件
1. 支持 plan[1] 模式
2. 支持 subagent[2] 模式
3. （推荐）subagent具有 独立上下文空间[3]
4. （推荐）主模型可以指定使用什么模型用于subagent
5. （可选）harness可以在subagent中，可以主动进行上下文缓存的写入；

## 前言
从agent时代，人们就意识到了上下文的重要性，从前大模型时期的上下文容量关注，到上下文工程时期的上下文结构质量关注，再到如今对整个agent所处在的harness环境的关注。

每次关于agent技术的进步，除了大语言模型能力边界的扩展外，另外一个十分重要，以至于可以说，我们对于上下文理解的进步，就是agent技术的进步。

在本篇分享中 我们将介绍一种*组合式*的agent模式，并在这个过程中希望为对您的开发有所帮助或启发。

## 这是什么？
Plan Tunneling 是一种将两种 agent 工作模式组合起来使用的 agent


主线程**全程待在 plan 模式**(只读、专注规划、上下文干净),
把需要"动手"的操作派给 **subagent** 去执行。subagent 不受 plan 模式约束,能真正读写、
跑命令、改文件;而执行产生的"脏上下文"留在 subagent 内,**不回流污染主线程**。


## 为什么有用

- **上下文不被污染**:执行细节(报错、输出、试错)封装在 subagent 里,主线程只拿结论。
- **不丢规划态**:一旦切到执行模式往往回不来,规划思路被打断;穿透模式让你一直在规划态里。
- **隔离与并行**:多个独立操作可派给多个 subagent 并行,互不干扰。

## 核心机制

| 角色 | 模式 | 职责 |
|------|------|------|
| 主线程 | plan(只读) | 规划、决策、给 subagent 下达明确任务、收口结论 |
| subagent | 可执行 | 真正落地操作,返回精炼结果,不把过程倒灌回主线程 |

## 适用场景

- 规划过程中需要"看一眼真实状态"或做一处小改,但不想退出规划。
- 想保持主线程上下文干净,只沉淀结论而非执行噪音。

## 注意

- 给 subagent 的任务要**明确、自包含**,并约定它只回精炼结果,避免脏上下文回流。
- 涉及不可逆/对外操作时,仍应先确认 —— 穿透的是模式壁垒,不是审慎本身。
